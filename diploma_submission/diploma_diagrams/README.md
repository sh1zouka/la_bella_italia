## 1. ER-диаграмма базы данных

**Файл:** `er/er_diagram.puml`
**Описание:** `er/er_description.txt`
**Рекомендуемый размер:** Половина листа A4

### Назначение
Отображает логическую структуру базы данных системы.
Показывает 4 таблицы: `users`, `orders`, `order_items`, `menu_items`.
Все таблицы связаны между собой внешними ключами.

### Связи между таблицами
```
users (1) ──── (N) orders (1) ──── (N) order_items (N) ──── (1) menu_items
```

### Ключевые особенности
- ✅ **Все таблицы связаны** — выполнено требование преподавателя
- ✅ Добавлена связующая таблица `order_items` для нормализации M:N связи
- ✅ JSONB-поля (macros, deliveryData, items) для гибкости
- ✅ Комментарии к каждой таблице и полю

## 2. Use Case диаграмма

**Файл:** `use_case/use_case.puml`
**Описание:** `use_case/use_case_description.txt`
**Рекомендуемый размер:** Половина листа A4

### Назначение
Описывает функциональные требования к системе с точки зрения
трёх типов пользователей (акторов).

1. **Гость** — неавторизованный пользователь (4 варианта)

- Управление корзиной и заказ
- История, профиль, статус

## 4. Диаграмма последовательности

**Файл:** `sequence/sequence_order.puml`
**Описание:** `sequence/sequence_description.txt`
**Рекомендуемый размер:** Целый лист A4 (альбомный)

### Назначение
Детально показывает временной порядок взаимодействия
компонентов при оформлении заказа.

### Этапы
1. Аутентификация (POST /api/login)
2. Просмотр меню (GET /api/menu)
3. Добавление в корзину (localStorage)
4. Оформление заказа (POST /api/orders) — **ключевой этап**
5. Обновление статуса (PATCH, админ)
6. Просмотр истории (GET /api/orders/my)

### Ключевой момент — расчёт бонусов на сервере:
```
total = Σ(price × qty)
bonusDiscount = min(bonusPoints, floor(total × 0.3))
finalTotal = total - bonusDiscount
earnedBonuses = floor(finalTotal × 0.05)
```

## 5. DFD диаграмма

**Файл:** `dfd/dfd.puml`
**Описание:** `dfd/dfd_description.txt`
**Рекомендуемый размер:** Целый лист A4

### Назначение
Моделирует потоки данных в системе на двух уровнях:
Level 0 (контекстная диаграмма) и Level 1 (детализация).

### Процессы Level 1
1. **Аутентификация и профиль** — вход, регистрация, сессии
2. **Управление меню** — просмотр, добавление, удаление
3. **Управление корзиной** — localStorage на клиенте
4. **Оформление и управление заказами** — бизнес-логика
5. **Администрирование системы** — управление и статистика

### Хранилища данных
D1 — Пользователи, D2 — Меню, D3 — Заказы,
D4 — Корзина (localStorage), D5 — Сессии (express-session)

## 6. Диаграмма развёртывания

**Файл:** `deployment/deployment.puml`
**Описание:** `deployment/deployment_description.txt`
**Рекомендуемый размер:** Половина листа A4

### Назначение
Показывает физическое распределение компонентов
по вычислительным узлам.

### Узлы
1. **Клиентский компьютер** — браузер (любая ОС)
2. **Сервер приложений** — Node.js (порт 3000)
3. **Сервер БД** — PostgreSQL 16 (порт 5432)

### Особенность
Два режима работы:
- `DB_ENGINE=postgres` — с PostgreSQL
- `DB_ENGINE=nedb` — встроенная БД (без установки PostgreSQL)

## 8. Диаграмма активности

**Файл:** `activity/activity_order.puml`
**Описание:** `activity/activity_description.txt`
**Рекомендуемый размер:** Целый лист A4 (книжный)

### Назначение
Моделирует поток управления процесса оформления заказа
от выбора блюд до получения заказа.

### Основные ветвления
1. Корзина пуста → вернуться в меню
2. Валидация провалена → подсветка ошибок
3. Не авторизован → перенаправление на вход
4. Ошибка БД → 500
5. Цикл изменения статуса заказа

### Статусы заказа
```
confirmed → preparing → delivering → done
                                    → cancelled
```
