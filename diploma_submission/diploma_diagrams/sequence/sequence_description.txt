============================================================
НАЗВАНИЕ ДИАГРАММЫ: Диаграмма последовательности
    оформления заказа
    (Sequence Diagram)
============================================================

ОПИСАНИЕ:
Диаграмма последовательности отображает временной порядок
взаимодействия объектов при оформлении заказа в системе
доставки «La Bella Italia». Показан полный жизненный цикл
заказа — от аутентификации до отслеживания статуса.

УЧАСТНИКИ ДИАГРАММЫ:
1. Пользователь (User) — инициирует действия
2. Браузер (Browser) — клиентская часть
3. Сервер (Express.js) — серверная логика
4. NeDB-обёртка (Wrapper) — слой абстракции БД
5. Sequelize ORM — объектно-реляционное отображение
6. PostgreSQL (БД) — база данных

ЭТАПЫ ВЗАИМОДЕЙСТВИЯ:

1. АУТЕНТИФИКАЦИЯ (POST /api/login)
   - Пользователь вводит email и пароль
   - Сервер проверяет rate limit (10 попыток за 15 мин)
   - bcrypt.compare() — сравнение хеша пароля
   - Создание серверной сессии (httpOnly cookie)
   - Ответ: данные пользователя (id, имя, роль, бонусы)

2. ПРОСМОТР МЕНЮ (GET /api/menu)
   - Запрос всех 50 блюд, отсортированных по id
   - Проход через NeDB-обёртку → Sequelize ORM → PostgreSQL
   - Возврат JSON с id, name, price, category, macros
   - Отображение карточек с БЖУ и кнопкой «В корзину»

3. ДОБАВЛЕНИЕ В КОРЗИНУ
   - Данные сохраняются в localStorage браузера
   - AJAX-запрос на сервер НЕ отправляется
   - После нажатия — визуальная обратная связь (анимация)

4. ОФОРМЛЕНИЕ ЗАКАЗА (POST /api/orders)
   - Самая важная часть диаграммы
   - Валидация полей доставки на клиенте
   - Расчёт на сервере:
     * total = Σ(price × quantity)
     * bonusDiscount = min(bonusPoints, floor(total × 0.3))
     * finalTotal = total - bonusDiscount
     * earnedBonuses = floor(finalTotal × 0.05)
   - Создание заказа в БД (INSERT INTO orders)
   - Обновление бонусов пользователя (UPDATE users)
   - Очистка корзины на клиенте

5. ОБНОВЛЕНИЕ СТАТУСА (админ, PATCH)
   - requireAdmin middleware проверяет роль
   - Изменение статуса: confirmed → preparing → delivering → done
   - UPDATE orders SET status = ?

6. ПРОСМОТР ИСТОРИИ (GET /api/orders/my)
   - Выборка всех заказов пользователя
   - Сортировка от новых к старым
   - Отображение статусов: Подтверждён, Готовится, В пути, Доставлен

ОСОБЕННОСТИ:
- Вся бизнес-логика (расчёт бонусов, валидация) выполняется
  на сервере — гарантирует целостность данных
- Административные действия защищены middleware requireAdmin
- Поток данных строго следует архитектуре: Browser → Express →
  NeDB-обёртка → Sequelize ORM → PostgreSQL и обратно
