# 📘 Сводка диаграмм для вставки в дипломный проект

**Проект:** Разработка веб-приложения для автоматизации приёма и
обработки заказов в итальянском ресторане «La Bella Italia»

---

## 1. ER-диаграмма базы данных
**(Entity-Relationship Diagram)**

*Рекомендуемое размещение: Глава 1.4 или 2.2*

### Описание для диплома:
ER-диаграмма отображает логическую структуру базы данных системы
доставки «La Bella Italia». База данных реляционного типа состоит
из четырёх таблиц, связанных между собой внешними ключами:

1. **users** (Пользователи) — хранит данные о зарегистрированных
   пользователях: уникальный идентификатор, имя, email (логин),
   хеш пароля (bcrypt), телефон, роль (user/admin), бонусные баллы
   и дату регистрации. Связь: один ко многим с таблицей orders.

2. **orders** (Заказы) — содержит информацию о заказах: номер заказа,
   внешний ключ на пользователя (userId), состав заказа (JSONB),
   суммы (total, bonusDiscount, finalTotal), начисленные бонусы,
   данные доставки (JSONB), статус и дату создания.

3. **order_items** (Позиции заказа) — связующая таблица для
   нормализации данных. Обеспечивает связь «многие ко многим»
   между заказами и блюдами меню. Хранит количество, цену
   и название блюда на момент заказа.

4. **menu_items** (Блюда меню) — хранит 50 блюд в 5 категориях
   (pizza, pasta, starters, desserts, drinks): название, описание,
   цену, вес, макросы (JSONB), флаг популярности.

**Связи между таблицами:**
- users (1) → orders (N) — один пользователь имеет много заказов
- orders (1) → order_items (N) — один заказ состоит из N позиций
- order_items (N) → menu_items (1) — N позиций ссылаются на 1 блюдо

Все таблицы связаны между собой, что обеспечивает ссылочную
целостность данных и возможность выполнения сложных
аналитических SQL-запросов.

## 2. Use Case диаграмма
**(Диаграмма вариантов использования)**

*Рекомендуемое размещение: Глава 1.3*

### Описание для диплома:
Диаграмма вариантов использования описывает функциональные
требования к системе с точки зрения трёх категорий пользователей
(акторов).

**Акторы системы:**
1. **Гость** — неавторизованный пользователь, который может
   просматривать меню, осуществлять поиск и фильтрацию блюд,
   а также зарегистрироваться или войти в систему.
2. **Клиент** — зарегистрированный пользователь, который помимо
   возможностей гостя может управлять корзиной, оформлять заказы
   с использованием бонусной системы, просматривать историю
   заказов, профиль и отслеживать статус текущего заказа.
3. **Администратор** — пользователь с расширенными правами,
   который может управлять меню (добавлять и удалять блюда),
   управлять заказами (изменять статус), просматривать список
   всех пользователей и статистику (выручка, средний чек).

**Всего 13 вариантов использования:** просмотр меню (UC-1),
поиск и фильтрация (UC-2), регистрация (UC-3), вход (UC-4),
управление корзиной (UC-5), оформление заказа (UC-6), история
заказов (UC-7), профиль (UC-8), статус заказа (UC-9), управление
меню (UC-10), управление заказами (UC-11), управление
пользователями (UC-12), статистика (UC-13).

## 3. Диаграмма архитектуры приложения
**(Architecture Diagram)**

*Рекомендуемое размещение: Глава 2.1*

### Описание для диплома:
Диаграмма архитектуры отображает трёхуровневую структуру
веб-приложения:

1. **Клиентский уровень (Frontend)** — браузер пользователя,
   выполняющий HTML-страницы, CSS-стили и JavaScript-модули.
   Корзина хранится в localStorage браузера для снижения
   нагрузки на сервер.

2. **Серверный уровень (Backend)** — Node.js с фреймворком
   Express.js. Включает middleware (helmet для безопасности,
   express-session для сессий, express-rate-limit для защиты
   от брутфорса, bcryptjs для хеширования паролей).
   NeDB-совместимая обёртка транслирует NeDB-стиль запросов
   в Sequelize ORM.

3. **Уровень данных (Database)** — PostgreSQL 16 с JSONB-полями
   для гибкого хранения неструктурированных данных.

**Поток данных:** Браузер → HTTP/JSON → Express.js → Middleware →
NeDB-обёртка → Sequelize ORM → PostgreSQL.

Архитектурной особенностью является NeDB-совместимая обёртка,
которая позволяет переключаться между PostgreSQL и встроенной
файловой БД NeDB без изменения кода приложения.

## 4. Диаграмма последовательности оформления заказа
**(Sequence Diagram)**

*Рекомендуемое размещение: Глава 2.4*

### Описание для диплома:
Диаграмма последовательности детально отображает временной
порядок взаимодействия компонентов системы при оформлении
заказа. Включает 6 этапов:

1. **Аутентификация** — пользователь вводит email и пароль,
   сервер проверяет rate limit (10 попыток за 15 минут),
   сравнивает хеш пароля (bcrypt) и создаёт серверную сессию.

2. **Просмотр меню** — браузер запрашивает GET /api/menu,
   данные проходят через NeDB-обёртку, Sequelize ORM и
   возвращаются из PostgreSQL в формате JSON.

3. **Добавление в корзину** — данные сохраняются в localStorage
   без отправки на сервер (архитектурное решение для снижения
   нагрузки).

4. **Оформление заказа (ключевой этап)** — после клиентской
   валидации полей (ФИО, телефон, адрес Москвы) отправляется
   POST /api/orders. Сервер рассчитывает: total = Σ(price × qty),
   bonusDiscount = min(bonusPoints, floor(total × 0.3)),
   finalTotal = total - bonusDiscount,
   earnedBonuses = floor(finalTotal × 0.05). Затем создаётся
   запись в таблице orders и обновляются бонусы пользователя.

5. **Обновление статуса (админ)** — администратор меняет статус
   через PATCH /api/admin/orders/:id, middleware requireAdmin
   проверяет права доступа.

6. **Просмотр истории** — пользователь получает список своих
   заказов через GET /api/orders/my.

## 5. DFD диаграмма
**(Data Flow Diagram — Level 0 и Level 1)**

*Рекомендуемое размещение: Глава 2.2*

### Описание для диплома:
Диаграмма потоков данных (DFD) моделирует движение данных
в системе на двух уровнях детализации.

**Уровень 0 (Контекстная диаграмма):**
Показывает систему как единый процесс, взаимодействующий
с тремя внешними сущностями (Гость, Клиент, Администратор)
и четырьмя хранилищами данных: Пользователи (D1), Меню (D2),
Заказы (D3), Корзина в localStorage (D4).

**Уровень 1 (Детализация):**
Декомпозирует систему на 5 процессов:
1. Аутентификация и профиль (1.0) — регистрация, вход, выход
2. Управление меню (2.0) — просмотр, добавление, удаление
3. Управление корзиной (3.0) — добавление/удаление на клиенте
4. Оформление и управление заказами (4.0) — создание, история
5. Администрирование системы (5.0) — заказы, пользователи,
   статистика

Каждый процесс имеет чётко определённые входные и выходные
потоки данных, а также связанные хранилища.

## 6. Диаграмма развёртывания
**(Deployment Diagram)**

*Рекомендуемое размещение: Глава 2.1*

### Описание для диплома:
Диаграмма развёртывания отображает физическую архитектуру
системы — распределение программных компонентов по трём
вычислительным узлам:

1. **Клиентский компьютер** — работает под любой современной
   ОС (Windows, macOS, Linux), использует браузер для доступа
   к приложению. Не требует установки дополнительного ПО.

2. **Сервер приложений** — Node.js 18+ с Express.js на порту 3000.
   Раздаёт статические файлы фронтенда и обрабатывает API-запросы.
   Поддерживает два режима работы: с PostgreSQL (DB_ENGINE=postgres)
   и со встроенной NeDB (DB_ENGINE=nedb) для демонстрации без
   установки СУБД.

3. **Сервер базы данных** — PostgreSQL 16 на порту 5432 с базой
   данных italian_delivery, содержащей таблицы users, menu_items
   и orders с JSONB-полями и индексами.

**Связи:** Клиент → HTTP (3000) → Сервер → TCP (5432) → БД.

## 7. Диаграмма классов
**(Class Diagram)**

*Рекомендуемое размещение: Глава 1.4 или 2.3*

### Описание для диплома:
Диаграмма классов отображает объектную модель приложения,
включающую три основных класса:

1. **User** (Пользователь) — атрибуты: id, name, email (уникальный),
   password (bcrypt-хеш), phone, role, bonusPoints, createdAt.
   Методы: register(), login(), getProfile(), updateBonuses(),
   getOrders(). Связь: один ко многим с классом Order.

2. **MenuItem** (Блюдо меню) — атрибуты: id, category (5 категорий),
   name, desc, price, weight, img, photo, macros (JSONB с КБЖУ),
   popular. Методы: findAll(), findByCategory(), search(), create(),
   deleteById().

3. **Order** (Заказ) — атрибуты: id, userId (FK), items (JSONB),
   total, bonusDiscount, finalTotal, earnedBonuses, deliveryData
   (JSONB), status (5 статусов), createdAt. Методы: create() с
   расчётом бонусов, findByUser(), findAll(), updateStatus(),
   deleteById(), getStats().

Связь между классами: User (1) → Order (*). MenuItem не имеет
прямой ассоциации с Order, так как состав заказа хранится
в JSONB-поле (документная модель внутри реляционной СУБД).

## 8. Диаграмма активности оформления заказа
**(Activity Diagram)**

*Рекомендуемое размещение: Глава 2.4*

### Описание для диплома:
Диаграмма активности моделирует поток управления процесса
оформления заказа от выбора блюд до получения заказа клиентом.
Включает следующие этапы:

1. **Выбор блюд** — пользователь просматривает меню, добавляет
   блюда в корзину. Если корзина пуста — возврат к меню.

2. **Ввод данных доставки** — пользователь вводит ФИО, телефон
   (маска +7) и адрес Москвы. Выполняется клиентская валидация
   с подсветкой ошибок.

3. **Отправка заказа** — POST /api/orders с проверкой авторизации.
   При отсутствии сессии — перенаправление на страницу входа.

4. **Серверная обработка** — расчёт сумм и бонусов, создание
   записи в БД, обновление бонусного баланса.

5. **Страница успеха** — отображение номера заказа, итоговой
   суммы и начисленных бонусов. Очистка корзины.

6. **Жизненный цикл заказа** — цикл изменения статуса
   (confirmed → preparing → delivering → done) под управлением
   администратора. Пользователь отслеживает статус в профиле.

На каждом шаге предусмотрена обработка ошибок с соответствующими
ветвлениями.

## 9. Диаграмма компонентов
**(Component Diagram)**

*Рекомендуемое размещение: Глава 2.1*

### Описание для диплома:
Диаграмма компонентов отображает структурную организацию
программных модулей системы и зависимости между ними.

**Клиентские компоненты:**
- 7 HTML-страниц (index, menu, cart, auth, profile, admin, 404)
- Единый файл стилей style.css
- 4 JavaScript модуля: db.js (API-клиент), auth.js (аутентификация),
  cart.js (корзина на localStorage), app.js (главная логика)

**Серверные компоненты:**
- server.js — Express.js с 14 API-эндпоинтами
- database.js — инициализация БД с NeDB-обёрткой
- 3 модели Sequelize: User.js, MenuItem.js, Order.js
- NeDB-интерфейс — слой абстракции БД

**База данных:** PostgreSQL 16 с таблицами users, menu_items, orders

**Связи:** HTML → CSS (стили), HTML → JS (скрипты), JS → localStorage
(данные), API (db.js) → Express.js (REST API), Express.js → NeDB API →
database.js → Sequelize → PostgreSQL.
