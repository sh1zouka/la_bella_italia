## 1. Обзор проекта

**Название:** Разработка веб-приложения для автоматизации приёма и обработки заказов в итальянском ресторане «La Bella Italia»

**Тип:** Выпускная квалификационная работа (ВКР)

**Цель:** Создать функциональное веб-приложение для автоматизации приёма и обработки заказов, которое позволяет ресторану отказаться от сторонних агрегаторов (Яндекс.Еда, Delivery Club), берущих комиссию ~30%.

**Основные возможности:**

| Для клиентов | Для администратора |
|:-------------|:-------------------|
| Просмотр меню с фильтрацией по категориям | Управление меню (добавление/удаление блюд) |
| Добавление в корзину | Управление заказами (смена статуса) |
| Оформление заказа с валидацией полей | Просмотр всех пользователей |
| Регистрация / авторизация | Статистика и аналитика |
| Бонусная программа (5% кэшбэк) | Выручка, средний чек |
| История заказов в профиле | Удаление заказов |
| 🎁 100 бонусов при регистрации | |

```
│   ↑                                         ↑       │
│   │        ┌──────────────────────┐         │       │
│   └────────┤  NeDB-совместимый API │─────────┘       │
│            │  (database.js)        │                 │
│            └──────────┬───────────┘                 │
│                       │                             │
│                       ▼                             │
│            ┌──────────────────────┐                 │
│            │     Sequelize ORM    │                 │
│            └──────────┬───────────┘                 │
└───────────────────────┼─────────────────────────────┘
                        │ SQL
                        ▼
           ┌──────────────────────────┐
           │      PostgreSQL          │
           │  Таблицы:                │
           │   • users                │
           │   • menu_items           │
           │   • orders               │
           └──────────────────────────┘
```

**Ключевой архитектурный паттерн:** NeDB-совместимая обёртка.
Серверный код написан так, будто использует NeDB (встроенную NoSQL БД), но на самом деле через `database.js` все запросы транслируются в Sequelize → PostgreSQL. Это позволило:
- Писать код на привычном NeDB-подобном API (`.find()`, `.insert()`, `.update()`, `.remove()`)
- Использовать мощную реляционную БД PostgreSQL
- Легко переключаться между БД без изменения кода

## 7. Как запустить сайт

### Шаг 1: Проверить PostgreSQL

Убедитесь, что PostgreSQL установлен и запущен:

```
pg_isready -h localhost
```

Если PostgreSQL не запущен (Windows):
```
net start postgresql-x64-16
```
*(цифра 16 может отличаться в зависимости от версии)*

Если PostgreSQL не установлен: скачайте с [postgresql.org](https://www.postgresql.org/download/windows/)

### Шаг 2: Создать базу данных

Перейдите в папку проекта и выполните:

```bash
cd C:\Users\costy\OneDrive\Desktop\final_diplom\italian-delivery
node setup_pg.js
```

Это создаст базы `italian_delivery` и `italian_delivery_test`.

**Если у вас другой пароль PostgreSQL**, отредактируйте в `database.js` строки:

```js
password: process.env.DB_PASSWORD || 'postgres',
```

### Шаг 3: Установить зависимости

```bash
npm install
```

### Шаг 4: Запустить сервер

```bash
npm start
# или
node server.js
```

**Сервер запустится на** → http://localhost:3000

### Шаг 5: Открыть в браузере

Перейдите по адресу: http://localhost:3000

## 11. Вопросы на защите — с ответами

### 🔹 1. Какая архитектура используется в приложении?

**Ответ:** Приложение построено по **клиент-серверной архитектуре** с использованием **REST API**. Клиент (браузер) отправляет HTTP-запросы к серверу Node.js + Express.js. Сервер обрабатывает запросы через единый API-слой. Данные хранятся в PostgreSQL, доступ к которым осуществляется через Sequelize ORM, но код написан через NeDB-совместимую обёртку, что упрощает разработку.

### 🔹 2. Почему вы выбрали NeDB-стиль вместо прямого Sequelize?

**Ответ:** NeDB предоставляет очень простой и интуитивный API — `.find()`, `.insert()`, `.update()`, `.remove()`. Я написал обёртку (`database.js`), которая принимает NeDB-стиль запросов и транслирует их в Sequelize. Это даёт два преимущества: (1) код получается компактнее и читаемее, (2) при необходимости можно легко переключиться на реальный NeDB (встроенная файловая БД) без переписывания кода.

### 🔹 3. Как обеспечивается безопасность?

**Ответ:**
- Пароли хешируются **bcryptjs** (10 раундов соли)
- Сессии хранятся на сервере (**express-session** с 7-дневной cookie)
- Защита от брутфорса — **express-rate-limit** (макс. 10 попыток входа за 15 минут)
- Все эндпоинты админ-панели защищены middleware `requireAdmin`, проверяющим роль
- Доступ к `/api/me` и `/api/orders/my` закрыт для неавторизованных (status 401)

### 🔹 4. Какие функции доступны администратору?

**Ответ:** Администратор имеет доступ к панели управления с 4 вкладками:
1. **Заказы** — просмотр всех заказов, смена статуса (Подтверждён → Готовится → В пути → Доставлен), удаление
2. **Пользователи** — список всех зарегистрированных пользователей с ролями и бонусами
3. **Меню** — добавление новых блюд, удаление существующих
4. **Статистика** — общая выручка, средний чек, количество заказов, распределение по статусам

### 🔹 5. Как работает бонусная система?

**Ответ:** При регистрации пользователь получает **100 бонусных баллов**. За каждый заказ начисляется **5% от суммы** в виде бонусов. Можно списать до **30% стоимости заказа** бонусами. Бонусы имеют уровни: Новичок (0-499), Гурман (500-1499), Шеф (1500+). В профиле отображается прогресс-бар до следующего уровня.

### 🔹 6. Какая база данных используется и почему?

**Ответ:** Используется **PostgreSQL** — одна из самых надёжных и распространённых реляционных СУБД. Доступ к данным организован через **Sequelize ORM**, который обеспечивает защиту от SQL-инъекций. Для упрощения кода используется NeDB-совместимая обёртка — это даёт лаконичный API без потери функциональности реляционной БД.

### 🔹 7. Как обрабатывается корзина? Где хранятся данные?

**Ответ:** Корзина хранится в **localStorage браузера** — это позволяет сохранять её при перезагрузке страницы и не нагружать сервер пустыми запросами. При оформлении заказа данные корзины отправляются на сервер, где создаётся запись в таблице `orders`. После успешного оформления корзина очищается.

### 🔹 8. Как происходит валидация полей заказа?

**Ответ:** На клиенте реализована **живая валидация** (по потере фокуса):
- **Имя и фамилия** — проверка на два слова (только буквы и дефис)
- **Телефон** — маска `+7 (XXX) XXX-XX-XX`, проверка на 11 цифр
- **Адрес доставки** — проверка на московский адрес (улица/проспект/бульвар + номер дома)
Поля подсвечиваются зелёным (OK) или красным (ошибка) с сообщением об ошибке.

### 🔹 9. Сколько тестов написано и что они проверяют?

**Ответ:** Написано **30+ автоматических тестов** с использованием Mocha, Chai и Supertest. Они проверяют:
- Публичное меню (4 теста) — структура, сортировка, категории
- Регистрацию (3 теста) — успех, ошибки, дубликаты

- Админ-панель (10+ тестов) — статистика, заказы, пользователи, меню, удаление
- 404 (2 теста)

### 🔹 10. Какие были сложности при разработке?

**Ответ:** Основные сложности:
1. **Защита от XSS и CSRF** — заявлена в дипломе. Используется **helmet()** для защиты HTTP-заголовков, httpOnly cookie с SameSite: strict. CSRF-токены не реализованы — это можно отметить как направление для улучшения.
2. **NeDB-обёртка** — требовалось аккуратно реализовать все цепочечные методы `.sort().limit()`, чтобы сохранить полную совместимость.
3. **Бонусная система** — нужно было корректно обработать кейсы: списание бонусов + начисление новых за тот же заказ, чтобы не было задвоений.
