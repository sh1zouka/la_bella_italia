📋 Обзор проекта

🎯 Назначение

Веб-приложение для автоматизации приёма и обработки заказов в итальянском ресторане «La Bella Italia». Позволяет ресторану отказаться от сторонних агрегаторов (Яндекс.Еда, Delivery Club), берущих комиссию ~30%.

👤 Для клиентов
  • Просмотр меню с поиском и категориями
  • Корзина с изменением количества
  • Оформление заказа с валидацией
  • Регистрация / авторизация
  • 🎁 Бонусная программа
  • 📋 История заказов в профиле
  • 📱 Адаптивный дизайн
🔧 Для администратора
  • 📦 Управление заказами (статусы, удаление)
  • 👥 Список пользователей
  • 🍕 CRUD меню (добавить/удалить блюда)
  • 📊 Статистика (выручка, средний чек)
  • 🔒 Защищённые админ-маршруты

🛠 Технологический стек

🔵 Сервер
  • Node.js
  • Express.js
  • express-session
  • bcryptjs
  • helmet
  • express-rate-limit
🟢 База данных
  • PostgreSQL (через Sequelize)
  • NeDB (встроенная — для демо без БД)
  • Два режима: DB_ENGINE=postgres | nedb
  • JSONB-поля (items, macros, deliveryData)
  • Sequelize ORM
🟣 Фронтенд + Тесты
  • HTML5 + CSS3 + Vanilla JS
  • Glassmorphism, анимации, skeleton
  • Адаптивная вёрстка
  • Mocha + Chai + Supertest
  • 45 тестов ✅
💡 Ключевая архитектурная особенность: Код написан через NeDB-совместимую обёртку (методы .find(), .insert(), .update(), .remove()), которая транслируется в Sequelize → PostgreSQL. Это позволяет легко переключаться между СУБД. А также — запускать проект без установки PostgreSQL!

💬 Вопросы на защите — с ответами

1. Какая архитектура используется в приложении?
Ответ: Клиент-серверная архитектура с REST API. Клиент (браузер) отправляет HTTP-запросы к серверу Node.js + Express.js. Сервер обрабатывает их через NeDB-совместимую обёртку, которая транслируется в Sequelize ORM → PostgreSQL. Фронтенд — SPA-like на Vanilla JS (без фреймворков).
2. Почему NeDB-обёртка, а не прямой Sequelize?
Ответ: NeDB предоставляет простой интуитивный API — .find(), .insert(), .update(), .remove(). Обёртка транслирует эти запросы в Sequelize. Это даёт: (1) компактный и читаемый код, (2) возможность переключиться на NeDB без переписывания кода, (3) единый интерфейс независимо от СУБД.
3. Как обеспечивается безопасность?
Ответ: Четыре уровня: (1) helmet — защита HTTP-заголовков; (2) bcryptjs — хеширование паролей (10 раундов); (3) express-rate-limit — 10 попыток входа за 15 минут; (4) middleware requireAdmin — проверка роли на всех админ-маршрутах. Сессии: httpOnly + SameSite:strict.
4. Какая БД используется и почему?
Ответ: PostgreSQL — надёжная реляционная СУБД. Через Sequelize ORM с защитой от SQL-инъекций. JSONB-поля в меню и заказах дают гибкость документной модели внутри реляционной БД.
5. Как работает бонусная система?
Ответ: 100 баллов при регистрации, 5% кэшбэк с каждого заказа, до 30% суммы можно оплатить бонусами. Есть защита от отрицательных значений (Math.max(0, ...)). Уровни: Новичок (0-499), Гурман (500-1499), Шеф (1500+).
6. Как хранится корзина?
Ответ: В localStorage браузера — не нагружает сервер, сохраняется при перезагрузке. При оформлении заказа данные отправляются на сервер, создаётся запись в таблице orders, корзина очищается.
7. Какие функции у администратора?
Ответ: Четыре вкладки: Заказы (просмотр, смена статуса, удаление), Пользователи (весь список), Меню (добавить/удалить блюда), Статистика (выручка, средний чек, количество заказов). Все админ-маршруты защищены ролью.
8. Сколько тестов и что они проверяют?
Ответ: 45 тестов (Mocha + Chai + Supertest). Проверяют: меню, регистрацию, вход/выход, профиль, создание заказов с бонусами, историю, админ-панель (статистика, пользователи, заказы, меню), удаление, 404. Все тесты проходят.
9. Какие были сложности?
Ответ: (1) NeDB-обёртка — нужно было реализовать цепочечные методы .sort().limit() для полной совместимости. (2) Бонусная система — корректная обработка списания + начисления за один заказ. (3) Валидация адреса Москвы — определение улицы и номера дома.
10. Почему проект не требует PostgreSQL для демо?
Ответ: Проект поддерживает два режима: DB_ENGINE=nedb (встроенная файловая БД, не требует установки) и DB_ENGINE=postgres. Переключение одной переменной окружения. Это сделано для удобства проверки комиссией.
11. Какие планы по развитию?
Ответ: Интеграция с платёжными системами, WebSockets для real-time обновлений, мобильное приложение, CSRF-защита, загрузка фото блюд, CRM и аналитика.
